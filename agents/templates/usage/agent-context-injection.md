# Agent 컨텍스트 주입 최적화

## 사용 시기

- Agent에게 프로젝트 컨텍스트를 효과적으로 전달해야 할 때
- 컨텍스트 윈도우 한계 내에서 정보 밀도를 최대화하고 싶을 때
- Agent가 반복적으로 동일한 실수를 하여 추가 맥락이 필요할 때
- 복잡한 프로젝트에서 Agent에게 제공할 정보를 선별해야 할 때

## 프롬프트

다음 프레임워크를 사용하여 Agent에게 주입할 컨텍스트를 설계하세요.

---

### 1. 컨텍스트 계층 구조

```
## 컨텍스트 설계

### Layer 1: 항상 포함 (Permanent Context)
<!-- 모든 대화에서 반드시 포함해야 하는 핵심 정보 -->

#### 프로젝트 정체성
- 프로젝트명: {{PROJECT_NAME}}
- 핵심 목적: {{PROJECT_PURPOSE}}
- 기술 스택: {{TECH_STACK}}

#### 불변 규칙
- {{INVARIANT_RULE_1}}
- {{INVARIANT_RULE_2}}
- {{?INVARIANT_RULE_3}}
<!-- 예: "모든 API 응답은 snake_case", "DB 직접 접근 금지" -->

### Layer 2: 작업별 포함 (Task Context)
<!-- 현재 작업에 직접 관련된 정보만 포함 -->

#### 관련 코드 영역
{{RELEVANT_CODE_AREAS}}
<!-- 현재 작업에 영향을 주는 파일/모듈 목록 -->

#### 관련 비즈니스 규칙
{{?BUSINESS_RULES}}
<!-- 작업에 영향을 주는 비즈니스 요구사항 -->

#### 최근 변경사항
{{?RECENT_CHANGES}}
<!-- 최근 관련 커밋이나 변경 내용 요약 -->

### Layer 3: 필요시 포함 (On-demand Context)
<!-- Agent가 요청하거나 특정 상황에서만 포함 -->

#### 히스토리
{{?HISTORY_CONTEXT}}
<!-- 이전 결정의 배경, 레거시 코드가 존재하는 이유 등 -->

#### 외부 의존성
{{?EXTERNAL_DEPENDENCIES}}
<!-- 외부 API 문서, 서드파티 라이브러리 동작 방식 등 -->

#### 디버깅 정보
{{?DEBUG_CONTEXT}}
<!-- 에러 로그, 스택 트레이스, 재현 단계 등 -->
```

### 2. 컨텍스트 압축 전략

```
## 압축 규칙

### 코드 컨텍스트 압축
<!-- 전체 파일 대신 관련 부분만 발췌 -->
원본 파일: {{SOURCE_FILE_PATH}}
포함할 범위: {{CODE_RANGE}}
압축 방식: {{COMPRESSION_METHOD}}
```

<!-- COMPRESSION_METHOD: 다음 중 선택:
- **함수 시그니처만**: 구현 생략, 인터페이스만 포함
- **핵심 로직만**: 보일러플레이트 제거, 비즈니스 로직만 포함
- **주석 요약**: 코드를 자연어 요약으로 대체
- **diff만**: 전체 파일 대신 변경 부분만 포함
-->

```
### 문서 컨텍스트 압축
<!-- 긴 문서를 핵심 요약으로 변환 -->
원본 문서: {{DOCUMENT_PATH}}
포함할 섹션: {{RELEVANT_SECTIONS}}
최대 길이: {{MAX_TOKEN_BUDGET}}
```

### 3. 컨텍스트 주입 포맷

```
## 최적 주입 형식

### XML 태그 활용
<project-context>
{{PERMANENT_CONTEXT}}
</project-context>

<task-context>
{{TASK_CONTEXT}}
</task-context>

<reference>
{{?ON_DEMAND_CONTEXT}}
</reference>
```

<!-- 태그로 감싸면 Agent가 컨텍스트의 종류와 중요도를 구분할 수 있음 -->

### 4. 컨텍스트 갱신 전략

```
## 갱신 규칙

### 갱신 트리거
- {{UPDATE_TRIGGER_1}}
- {{UPDATE_TRIGGER_2}}
<!-- 예: "새 파일이 추가되었을 때", "아키텍처 결정이 변경되었을 때" -->

### 갱신 절차
1. {{UPDATE_STEP_1}}
2. {{UPDATE_STEP_2}}
3. {{UPDATE_STEP_3}}

### 버전 관리
- 현재 컨텍스트 버전: {{?CONTEXT_VERSION}}
- 마지막 갱신일: {{?LAST_UPDATED}}
```

### 컨텍스트 효율성 체크

주입 전에 다음을 확인하세요:

| 검증 항목 | 기준 |
|-----------|------|
| 정보 밀도 | 모든 문장이 Agent의 행동에 영향을 주는가? |
| 중복 제거 | 동일 정보가 여러 곳에 반복되지 않는가? |
| 최신성 | 포함된 정보가 현재 코드베이스를 반영하는가? |
| 관련성 | 현재 작업과 무관한 정보가 포함되지 않았는가? |
| 크기 | 전체 컨텍스트가 토큰 예산 내에 있는가? |

## 품질 기준

- [ ] 컨텍스트가 3개 계층(항상/작업별/필요시)으로 분류되었는가?
- [ ] 각 계층의 포함 기준이 명확한가?
- [ ] 불필요한 정보가 제거되었는가? (과잉 컨텍스트 방지)
- [ ] 코드 컨텍스트에 적절한 압축 전략이 적용되었는가?
- [ ] XML 태그 등으로 컨텍스트 종류가 구분되었는가?
- [ ] 컨텍스트 갱신 트리거가 정의되었는가?
- [ ] 전체 토큰 사용량이 예산 내인가?
