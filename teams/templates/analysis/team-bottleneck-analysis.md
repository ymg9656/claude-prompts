### 분석 프레임워크

다음 5가지 병목 유형을 체계적으로 검사하세요:

#### 유형 1: 자원 병목 (Resource Bottleneck)

특정 멤버에게 작업이 과도하게 집중되는 경우.

분석 항목:
- 멤버별 작업 할당량 분포
- 멤버별 평균 작업 시간
- 유휴 멤버와 과부하 멤버 식별

```
[분석 결과 형식]
멤버별 작업 부하:
  member-a: ████████████ (12 tasks) ← 과부하
  member-b: ████ (4 tasks)
  member-c: ██ (2 tasks) ← 유휴
```

#### 유형 2: 의존성 병목 (Dependency Bottleneck)

작업 간 의존 관계로 인해 대기 시간이 발생하는 경우.

분석 항목:
- 크리티컬 패스 길이
- 의존성 체인에서 가장 긴 대기 시간
- 불필요한 순차 의존성 식별

#### 유형 3: 커뮤니케이션 병목 (Communication Bottleneck)

메시지 전달/응답 지연으로 작업이 정체되는 경우.

분석 항목:
- 메시지 응답 시간 분포
- 재작업 요청 빈도 (품질 커뮤니케이션 부족 지표)
- 정보 부족으로 인한 블로커 발생 횟수

#### 유형 4: 페이즈 전환 병목 (Phase Transition Bottleneck)

페이즈 간 핸드오프에서 지연이 발생하는 경우.

분석 항목:
- 페이즈 완료 후 다음 페이즈 시작까지의 대기 시간
- 핸드오프 시 누락되는 정보
- 동기화 대기 시간

#### 유형 5: 품질 병목 (Quality Bottleneck)

리뷰/검증 단계에서 반복적으로 반려되는 경우.

분석 항목:
- 첫 통과율 (first-pass rate)
- 반려 사유 분류 (기능 미달 / 코드 품질 / 테스트 부족 등)
- 평균 수정 반복 횟수
