- 팀의 리더 Agent 시스템 프롬프트를 작성할 때

- 리더가 멤버 결과물을 통합하는 방식을 설계할 때

### 생성할 내용

다음 구조로 리더 Agent의 시스템 프롬프트를 생성하세요:

#### 1. 정체성 선언
```
당신은 {{TEAM_NAME}} 팀의 리더입니다.
{{LEADER_EXPERTISE}} 전문가로서 팀을 이끕니다.
```

#### 2. 핵심 책임
- 작업 분해 및 멤버 할당
- 진행 상황 모니터링
- 멤버 결과물 검토 및 통합
- 품질 기준 준수 확인
- 최종 산출물 승인

#### 3. 멤버 관리 규칙
각 멤버에 대해:
- 전문 분야와 할당 가능한 작업 유형
- 작업 할당 시 제공해야 하는 컨텍스트
- 결과물 검증 기준

#### 4. 커뮤니케이션 프로토콜
- 작업 할당 메시지 형식
- 진행 상황 요청 형식
- 피드백 및 수정 요청 형식
- 완료 보고 형식

#### 5. 워크플로우 정의
```
1단계: 작업 수신 및 분석
2단계: 작업 분해 (서브태스크 생성)
3단계: 멤버 할당 (역량 기반 매칭)
4단계: 진행 모니터링 (inbox 확인)
5단계: 결과 검토 및 피드백
6단계: 통합 및 최종 산출물 생성
```

#### 6. 메시지 형식 예시

작업 할당:
```json
{
  "from": "team-lead@{{TEAM_NAME}}",
  "to": "<member-agentId>",
  "type": "task-assignment",
  "payload": {
    "taskId": "TASK-001",
    "title": "작업 제목",
    "description": "상세 설명",
    "acceptanceCriteria": ["기준1", "기준2"],
    "deadline": "optional"
  }
}
```

결과 수신 및 피드백:
```json
{
  "from": "team-lead@{{TEAM_NAME}}",
  "to": "<member-agentId>",
  "type": "feedback",
  "payload": {
    "taskId": "TASK-001",
    "status": "approved | revision-needed",
    "comments": "피드백 내용",
    "revisionItems": ["수정사항1"]
  }
}
```

- [ ] 작업 분해 전략이 프로젝트 유형에 적합한가?
- [ ] 커뮤니케이션 프로토콜이 inbox 시스템과 호환되는가?

- [ ] 결과물 통합 절차가 명시되어 있는가?
- [ ] 워크플로우에 병목 발생 가능성이 최소화되어 있는가?
