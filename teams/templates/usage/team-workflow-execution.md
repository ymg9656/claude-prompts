# 팀 워크플로우 실행

## 사용 시기

- 분배된 작업을 팀 단위로 실행하고 관리할 때
- 멀티 페이즈 워크플로우의 진행 상황을 추적할 때
- 워크플로우 실행 중 발생하는 이슈에 대응할 때

### 실행 관리 프로토콜

각 페이즈에 대해 다음 절차를 수행하세요:

#### 1. 페이즈 시작

해당 페이즈의 참여 멤버에게 작업을 할당합니다:

```json
{
  "from": "team-lead@{{TEAM_NAME}}",
  "to": "<member-agentId>",
  "type": "phase-start",
  "payload": {
    "phaseId": "PHASE-X",
    "phaseName": "페이즈 이름",
    "input": "이전 페이즈 결과 또는 초기 입력",
    "tasks": ["할당된 작업 목록"],
    "completionCriteria": ["완료 기준"]
  }
}
```

#### 2. 진행 상황 모니터링

각 멤버의 inbox를 주기적으로 확인하고 상태를 추적합니다:

```
진행 상황 대시보드:
┌─────────────┬──────────┬──────────┬──────────┐
│ Phase       │ 멤버     │ 상태     │ 진행률   │
├─────────────┼──────────┼──────────┼──────────┤
│ Phase 1     │ member-a │ 진행 중  │ 60%      │
│ Phase 1     │ member-b │ 완료     │ 100%     │
│ Phase 2     │ member-c │ 대기     │ 0%       │
└─────────────┴──────────┴──────────┴──────────┘
```

#### 3. 이슈 대응

이슈 발생 시 유형별 대응 전략:

- **블로커 (blocked)**: 원인 분석 후 다른 멤버에게 지원 요청 또는 작업 재할당
- **도움 요청 (needs-help)**: 관련 전문가 멤버를 연결하거나 리더가 직접 지원
- **품질 미달 (revision-needed)**: 구체적 피드백과 함께 수정 요청
- **의존성 지연**: 크리티컬 패스 재계산 및 병렬화 가능 작업 식별

#### 4. 페이즈 완료 및 전환

페이즈 완료 확인 후 다음 페이즈로 전환합니다:

```json
{
  "from": "team-lead@{{TEAM_NAME}}",
  "type": "phase-transition",
  "payload": {
    "completedPhase": "PHASE-X",
    "nextPhase": "PHASE-Y",
    "summary": "완료된 페이즈 결과 요약",
    "handoff": {
      "artifacts": ["전달할 산출물"],
      "context": "다음 페이즈에 필요한 컨텍스트"
    }
  }
}
```

#### 5. 워크플로우 완료

모든 페이즈 완료 후 최종 보고서를 생성합니다:

```markdown
## 워크플로우 실행 결과

### 요약
- 워크플로우: {{WORKFLOW_NAME}}
- 총 페이즈: N개
- 총 소요 시간: 예상 vs 실제
- 참여 멤버: N명

### 페이즈별 결과
| Phase | 상태 | 산출물 | 비고 |
|-------|------|--------|------|
| Phase 1 | 완료 | ... | ... |

### 최종 산출물
- [산출물 목록 및 위치]

### 발견된 이슈 및 교훈
- [이슈와 해결 방법]
```
