# 에이전트 간 통신 설계

## 사용 시기

- 팀 멤버 간 메시지 패싱 프로토콜을 설계할 때
- inboxes/ 디렉토리 기반 통신 구조를 정의할 때
- 구독(subscription) 패턴과 알림 전략을 수립할 때

### 설계할 내용

#### 1. inboxes/ 디렉토리 구조

팀의 inbox 구조를 설계하세요:

```
inboxes/
├── team-lead@{{TEAM_NAME}}.json
├── member-a@{{TEAM_NAME}}.json
├── member-b@{{TEAM_NAME}}.json
└── ...
```

각 inbox 파일의 메시지 형식:
```json
{
  "messages": [
    {
      "id": "msg-uuid",
      "from": "sender-agentId",
      "to": "receiver-agentId",
      "type": "메시지 유형",
      "timestamp": 1770709790101,
      "payload": {},
      "read": false
    }
  ]
}
```

#### 2. 메시지 유형 정의

다음 메시지 유형을 정의하세요:

| 유형 | 발신자 | 수신자 | 용도 |
|------|--------|--------|------|
| `task-assignment` | 리더 | 멤버 | 작업 할당 |
| `task-result` | 멤버 | 리더 | 작업 결과 보고 |
| `feedback` | 리더 | 멤버 | 피드백/수정 요청 |
| `help-request` | 멤버 | 리더 또는 다른 멤버 | 도움 요청 |
| `status-update` | 멤버 | 리더 | 진행 상황 보고 |
| `broadcast` | 리더 | 전체 | 공지/전달 사항 |
| `handoff` | 멤버 | 다음 멤버 | 작업 인수인계 |

#### 3. 구독 패턴 설계

각 멤버의 `subscriptions` 배열을 설계하세요:

{{WORKFLOW_TYPE}}에 따른 패턴:

**hub-and-spoke (리더 중심)**:
```
리더: [모든 멤버 구독]
멤버: [리더만 구독]
```

**mesh (자유 통신)**:
```
모든 멤버: [관련 멤버 상호 구독]
```

**pipeline (순차 전달)**:
```
멤버 A: [리더 구독]
멤버 B: [멤버 A, 리더 구독]
멤버 C: [멤버 B, 리더 구독]
```

#### 4. 메시지 흐름 다이어그램

워크플로우에 따른 메시지 흐름을 텍스트 다이어그램으로 표현:

```
[리더] --task-assignment--> [멤버A]
[리더] --task-assignment--> [멤버B]
[멤버A] --task-result--> [리더]
[멤버A] --handoff--> [멤버C]
[멤버B] --task-result--> [리더]
[멤버C] --task-result--> [리더]
[리더] --broadcast--> [전체: 완료 알림]
```

#### 5. 에러 처리 프로토콜

통신 실패 및 에러 상황에 대한 처리:

- **메시지 전달 실패**: 재시도 정책 정의
- **응답 지연**: 타임아웃 기준과 에스컬레이션 경로
- **충돌 해결**: 동일 리소스에 대한 동시 수정 시 처리 방법
- **비정상 종료**: 멤버 Agent 비정상 종료 시 작업 재할당 절차
