# Project Planning Skill

프로젝트의 요구사항을 분석하고 기능 명세서, WBS(작업 분해 구조), 일정 계획을 작성하는 Skill입니다. 기획 단계에서 "무엇을 만들 것인가"와 "어떤 순서로 진행할 것인가"를 체계적으로 정의합니다.

## 핵심 기능

- 요구사항 분석 및 구조화 (사용자 요구 → 기능 요구사항 → 기술 요구사항)
- 기능 명세서 자동 생성 (화면별 기능, 입출력 스펙, 엣지 케이스)
- WBS 작성 (작업 분해, 의존성 정의, 담당자 매핑)
- 일정 산정 (작업별 공수 추정, 크리티컬 패스 식별)
- 마일스톤 정의 및 진행 추적 체크리스트 생성

## 워크플로우

### 1. 프로젝트 현황 파악

기존 프로젝트 구조와 문서를 분석합니다.

```bash
# 프로젝트 구조 확인
ls -la src/ docs/ 2>/dev/null

# 기존 기획/요구사항 문서 확인
ls docs/*.md docs/planning/ docs/requirements/ 2>/dev/null

# 기존 이슈/태스크 확인
gh issue list --limit 50 2>/dev/null
```

### 2. 요구사항 분석

요구사항을 수집하고 구조화합니다.

#### 요구사항 명세서 형식

```markdown
# 요구사항 명세서

## 프로젝트 개요
- 프로젝트명:
- 목적:
- 대상 사용자:

## 기능 요구사항

### FR-001: 기능 제목
- 우선순위: P0 (필수) | P1 (중요) | P2 (선택)
- 설명: 기능 상세 설명
- 사용자 시나리오: 사용자가 ~하면 시스템이 ~한다
- 입력: 필요한 입력 데이터
- 출력: 기대 결과
- 엣지 케이스: 예외 상황 목록
- 관련 화면: 화면 ID 또는 이름

## 비기능 요구사항
- 성능: 응답 시간, 동시 접속자 등
- 보안: 인증/인가, 데이터 암호화 등
- 확장성: 예상 트래픽 증가 대응
```

### 3. 기능 명세서 작성

기능별 상세 명세를 작성합니다.

#### 기능 명세서 형식

```markdown
# 기능 명세서: {기능명}

## 개요
- 기능 ID: FR-XXX
- 관련 화면: 화면 ID
- 담당자: (팀 생성 시 배정)

## 화면 흐름
1. 사용자가 ~에 접근
2. 시스템이 ~를 표시
3. 사용자가 ~를 입력/선택
4. 시스템이 ~를 처리

## API 스펙 (초안)
- Method: GET | POST | PUT | DELETE
- Path: /api/resource
- Request Body: { field: type }
- Response: { field: type }
- 에러 응답: { code, message }

## 비즈니스 규칙
- 규칙 1: 조건 → 처리
- 규칙 2: 조건 → 처리

## 엣지 케이스
| 상황 | 기대 동작 |
|------|----------|
| 입력값 누락 | 오류 메시지 표시 |
| 권한 없음 | 403 에러 |

## 데이터 모델 (초안)
- 엔티티: 필드 목록
- 관계: 1:N, N:M 등
```

### 4. WBS 작성

작업을 분해하고 일정을 산정합니다.

#### WBS 형식

```markdown
# WBS (Work Breakdown Structure)

## 마일스톤

| 마일스톤 | 목표일 | 산출물 |
|---------|--------|--------|
| M1: 기획 완료 | W1 | 요구사항 명세서, 기능 명세서 |
| M2: 설계 완료 | W2 | API 스펙, DB 스키마, 화면 설계 |
| M3: 개발 완료 | W4 | 백엔드 API, 프론트엔드 UI |
| M4: 테스트 완료 | W5 | 테스트 리포트, 버그 수정 |
| M5: 배포 | W6 | 프로덕션 릴리스 |

## 작업 목록

| ID | 작업 | 담당 | 선행 작업 | 공수(일) | 상태 |
|----|------|------|----------|---------|------|
| T-001 | 요구사항 분석 | planner | — | 2 | 대기 |
| T-002 | 기능 명세서 작성 | planner | T-001 | 3 | 대기 |
| T-003 | API 설계 | backend | T-002 | 2 | 대기 |
| T-004 | DB 스키마 설계 | backend | T-002 | 1 | 대기 |
| T-005 | UI 설계 | frontend | T-002 | 2 | 대기 |

## 크리티컬 패스
T-001 → T-002 → T-003 → T-006 (API 구현) → T-009 (통합 테스트)
```

### 5. 문서 관리

프로젝트 전체 문서 체계를 수립합니다.

```markdown
# 프로젝트 문서 구조

docs/
├── planning/
│   ├── requirements.md        # 요구사항 명세서
│   ├── feature-specs/         # 기능별 명세서
│   │   ├── FR-001-login.md
│   │   ├── FR-002-dashboard.md
│   │   └── ...
│   ├── screen-flow.md         # 화면 흐름도
│   └── data-model.md          # 데이터 모델 초안
├── management/
│   ├── wbs.md                 # WBS / 일정 계획
│   ├── risk-register.md       # 리스크 레지스터
│   └── progress/              # 진행 보고서
│       ├── week-01.md
│       └── ...
├── technical/                 # (각 담당자가 작성)
│   ├── api-spec.md
│   ├── db-schema.md
│   └── architecture.md
└── changelog.md               # 변경 이력
```

## 에러 처리

| 에러 | 원인 | 해결 |
|------|------|------|
| 요구사항 모호 | 이해관계자 입력 부족 | 구체적 질문 목록 작성하여 확인 요청 |
| 범위 초과 (Scope Creep) | 기획 중 기능 추가 요청 | 우선순위 재평가, P0/P1/P2 분류 후 조정 |
| 일정 초과 | 공수 과소 추정 | 버퍼 포함한 재산정, 범위 축소 검토 |
| 문서 불일치 | 여러 문서 간 정보 불일치 | 변경 이력 관리, 단일 소스(Single Source of Truth) 유지 |
| 의존성 누락 | 작업 간 선행 관계 미파악 | 의존성 그래프 재검토, 팀원에게 교차 확인 |

## 사용 예시

### 예시 1: 요구사항 분석

```
사용자: "사용자 대시보드 기능을 기획해줘"

→ Skill이 프로젝트 분석 후:
  "대시보드 기능 요구사항을 분석했습니다.

  기능 요구사항 (7개):
  - FR-001: 대시보드 메인 뷰 (P0)
  - FR-002: 실시간 통계 위젯 (P0)
  - FR-003: 기간별 차트 (P1)
  - FR-004: 위젯 커스터마이징 (P2)
  - FR-005: 대시보드 공유 (P2)
  - FR-006: 데이터 내보내기 (P1)
  - FR-007: 모바일 반응형 (P1)

  비기능 요구사항:
  - 대시보드 로딩 2초 이내
  - 실시간 데이터 30초 주기 갱신

  docs/planning/requirements.md에 저장했습니다.
  미결정 사항 2건이 있습니다."
```

### 예시 2: WBS 생성

```
사용자: "이 기능의 WBS 만들어줘"

→ Skill이 기능 명세서 분석 후:
  "WBS를 생성했습니다.

  마일스톤 4개, 작업 18개 도출
  총 예상 공수: 23일

  크리티컬 패스:
  요구사항 분석(2일) → 기능 명세(3일) → API 설계(2일)
  → API 구현(5일) → 통합 테스트(3일) = 15일

  병렬 가능 작업:
  - UI 설계 + API 설계 (동시 진행 가능)
  - 백엔드 구현 + 프론트엔드 구현 (API 설계 완료 후)

  docs/management/wbs.md에 저장했습니다."
```

### 예시 3: 기능 명세서 작성

```
사용자: "로그인 기능 명세서 작성해줘"

→ Skill이 기존 코드/문서 분석 후:
  "로그인 기능 명세서를 작성했습니다.

  화면 흐름: 4단계
  API 엔드포인트: 3개 (로그인, 토큰 갱신, 로그아웃)
  비즈니스 규칙: 5개
  엣지 케이스: 8개 (비밀번호 오류, 계정 잠금, 세션 만료 등)

  docs/planning/feature-specs/FR-001-login.md에 저장했습니다."
```
