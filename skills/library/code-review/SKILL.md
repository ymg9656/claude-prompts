# Code Review Skill

코드 변경사항에 대해 체계적인 리뷰를 수행하는 Skill입니다. 가독성, SOLID 원칙, 보안, 성능, 에러 핸들링 관점에서 코드를 분석하고, 발견된 이슈를 심각도별로 분류하여 개선 방안을 제시합니다.

## 핵심 기능

- 파일별 코드 리뷰 체크리스트 자동 실행
- 가독성, SOLID 원칙, 보안, 성능, 에러 핸들링 5가지 관점 분석
- 이슈 심각도 분류 (Critical / Major / Minor / Info)
- PR diff 기반 변경사항 집중 리뷰
- 언어별 베스트 프랙티스 적용 (TypeScript, Python, Go, Java 등)

## 워크플로우

### 1. 변경사항 수집

리뷰 대상 코드의 변경사항을 수집합니다.

```bash
# PR 기반 변경 파일 목록 확인
git diff --name-only main...HEAD

# 변경사항 상세 확인
git diff main...HEAD
```

### 2. 파일별 분석

각 변경 파일에 대해 5가지 관점으로 분석합니다.

```bash
# 변경된 파일 내용 확인
git diff main...HEAD -- src/services/user.service.ts

# 파일 전체 컨텍스트 확인 (변경 부분의 주변 코드)
cat src/services/user.service.ts
```

#### 리뷰 체크리스트

**가독성**
- 함수/변수 이름이 의도를 명확히 전달하는가
- 함수 길이가 적절한가 (30줄 이하 권장)
- 중복 코드가 없는가
- 주석이 "왜"를 설명하는가 ("무엇"이 아닌)

**SOLID 원칙**
- 단일 책임 원칙: 클래스/함수가 하나의 역할만 수행하는가
- 개방-폐쇄 원칙: 확장에 열려 있고 수정에 닫혀 있는가
- 의존성 역전: 구체 클래스가 아닌 추상에 의존하는가

**보안**
- SQL 인젝션, XSS 등 인젝션 취약점이 없는가
- 인증/인가 검증이 적절한가
- 민감 정보(비밀번호, 토큰)가 로그에 노출되지 않는가
- 입력값 유효성 검증이 수행되는가

**성능**
- N+1 쿼리 문제가 없는가
- 불필요한 데이터 로딩이 없는가
- 적절한 인덱스를 활용하는가
- 메모리 누수 가능성이 없는가

**에러 핸들링**
- 예외 처리가 적절한가
- 에러 메시지가 사용자 친화적인가
- 실패 시 리소스가 정리되는가 (try-finally, using 등)

### 3. 이슈 분류

발견된 이슈를 심각도별로 분류합니다.

| 심각도 | 기준 | 조치 |
|--------|------|------|
| Critical | 보안 취약점, 데이터 손실 가능성 | 즉시 수정 필수 |
| Major | 버그, 성능 문제, SOLID 위반 | 머지 전 수정 권장 |
| Minor | 네이밍, 코드 스타일, 경미한 중복 | 개선 권장 |
| Info | 제안, 대안 접근법, 학습 포인트 | 참고 사항 |

### 4. 리뷰 리포트 생성

분석 결과를 종합하여 리뷰 리포트를 작성합니다.

```markdown
## 코드 리뷰 결과

### 요약
- 리뷰 파일: N개
- 발견 이슈: Critical X개, Major Y개, Minor Z개, Info W개

### 이슈 목록
#### [Critical] 파일명:라인 - 이슈 제목
설명 및 수정 제안

#### [Major] 파일명:라인 - 이슈 제목
설명 및 수정 제안
```

## 에러 처리

| 에러 | 원인 | 해결 |
|------|------|------|
| diff가 비어 있음 | 비교 대상 브랜치 지정 오류 | 올바른 base 브랜치 확인 (`main`, `develop` 등) |
| 파일을 읽을 수 없음 | 삭제된 파일 또는 바이너리 파일 | 텍스트 파일만 리뷰 대상으로 필터링 |
| 리뷰 범위 초과 | 변경 파일이 너무 많음 (50개 이상) | 핵심 파일 우선순위 지정 후 단계적 리뷰 |
| 언어 미지원 | 분석할 수 없는 파일 형식 | 지원 언어 목록 안내 및 일반 리뷰 수행 |

## 사용 예시

### 예시 1: PR 전체 리뷰

```
사용자: "이 PR 코드 리뷰 해줘"

→ Skill이 변경사항 분석 후:
  "코드 리뷰 결과:

  리뷰 파일: 8개 | 이슈: Critical 1, Major 3, Minor 5

  [Critical] src/auth/login.service.ts:42
  SQL 인젝션 취약점: 사용자 입력이 직접 쿼리에 삽입됨
  수정: parameterized query 또는 ORM 메서드 사용

  [Major] src/services/order.service.ts:78
  N+1 쿼리: 주문 목록 조회 시 각 주문별로 상품 쿼리 발생
  수정: eager loading 또는 join 쿼리 사용

  [Major] src/controllers/user.controller.ts:23
  에러 핸들링 누락: 비동기 함수에 try-catch 없음
  수정: 글로벌 에러 핸들러 또는 try-catch 추가"
```

### 예시 2: 특정 파일 집중 리뷰

```
사용자: "payment.service.ts 보안 관점으로 리뷰해줘"

→ Skill이 보안 체크리스트 실행 후:
  "보안 리뷰 결과 (payment.service.ts):

  [Critical] 라인 56: 카드 번호가 로그에 평문으로 기록됨
  수정: 마스킹 처리 (****-****-****-1234)

  [Major] 라인 89: 결제 금액 유효성 검증 누락
  수정: 금액이 0 이하이거나 최대값 초과 시 거부

  [Minor] 라인 112: 하드코딩된 API 키
  수정: 환경 변수로 이동"
```

### 예시 3: 자동 리뷰 요약

```
사용자: "변경사항 리뷰하고 요약해줘"

→ Skill이 전체 분석 후:
  "전체적으로 잘 작성된 코드입니다.

  개선 필요 사항:
  1. UserService의 메서드가 7개의 책임을 갖고 있어 분리 필요 (SRP 위반)
  2. 에러 응답 형식이 일관되지 않음 — 공통 에러 포맷 적용 권장
  3. 테스트 커버리지가 추가된 코드에 대해 부족함

  긍정적 사항:
  - 타입 정의가 명확하고 일관됨
  - DTO 검증 로직이 잘 분리되어 있음
  - 커밋 메시지가 변경사항을 잘 설명함"
```
