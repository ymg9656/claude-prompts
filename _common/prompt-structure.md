# 프롬프트 표준 구조

모든 템플릿이 따르는 5단계 구조입니다.

---

## 1단계: 역할 정의 (Identity)

프롬프트 수행 주체의 정체성을 명확히 합니다.

```markdown
당신은 {{ROLE}}입니다.
{{?EXPERTISE}} 분야의 전문가로서 행동합니다.
```

**핵심 원칙:**
- 구체적인 역할명을 사용 (예: "시니어 백엔드 개발자" > "개발자")
- 전문 분야를 1~2개로 한정
- 행동 방식의 톤 명시 (예: "간결하고 실용적으로")

## 2단계: 컨텍스트 설정 (Context)

작업 환경과 배경 정보를 제공합니다.

```markdown
## 프로젝트 배경
- 프로젝트: {{PROJECT_NAME}}
- 기술 스택: {{TECH_STACK}}
- 현재 상태: {{CURRENT_STATE}}

## 제약 조건
- {{CONSTRAINT_1}}
- {{CONSTRAINT_2}}
```

**핵심 원칙:**
- 필요한 정보만 포함 (과잉 컨텍스트 주의)
- 기술적 제약과 비즈니스 제약 분리
- 현재 상태를 객관적으로 기술

## 3단계: 작업 지시 (Task)

수행할 작업을 명확하게 정의합니다.

```markdown
## 작업
다음을 수행하세요:

1. {{STEP_1}}
2. {{STEP_2}}
3. {{STEP_3}}

## 입력
{{INPUT_DATA}}

## 출력 형식
{{OUTPUT_FORMAT}}
```

**핵심 원칙:**
- 단계별 분리 (한 번에 하나의 작업)
- 입력과 출력 형식 명시
- 모호한 동사 대신 구체적 동사 사용 ("개선하세요" → "성능 병목을 식별하고 해결 방안을 제안하세요")

## 4단계: 예시 제공 (Examples)

기대하는 결과물의 형태를 보여줍니다.

```markdown
## 예시

### 좋은 예시
{{GOOD_EXAMPLE}}

### 나쁜 예시
{{BAD_EXAMPLE}}
<!-- 이유: {{REASON}} -->
```

**핵심 원칙:**
- 최소 1개의 좋은 예시 필수
- 나쁜 예시로 안티패턴 명시
- 예시가 실제 상황과 유사해야 함

## 5단계: 품질 기준 (Quality Criteria)

결과물 검증 체크리스트를 제공합니다.

```markdown
## 품질 기준

- [ ] {{CRITERION_1}}
- [ ] {{CRITERION_2}}
- [ ] {{CRITERION_3}}
```

**핵심 원칙:**
- 검증 가능한 기준만 포함
- 우선순위 순서로 나열
- 3~7개 항목 권장

---

## 전체 템플릿 스켈레톤

```markdown
## 사용 시기

<!-- 이 템플릿을 선택해야 하는 상황을 설명합니다 -->

## 프롬프트

당신은 {{ROLE}}입니다.

### 컨텍스트
{{CONTEXT}}

### 작업
{{TASK_DESCRIPTION}}

### 예시
{{EXAMPLE}}

## 품질 기준

- [ ] 기준 1
- [ ] 기준 2
- [ ] 기준 3
```
