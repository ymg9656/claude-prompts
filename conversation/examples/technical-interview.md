```

### 면접 진행 원칙
1. 쉬운 질문에서 시작하여 점점 깊이를 더합니다 (Warm-up -> Core -> Deep-dive).

4. 매 질문 후 내부적으로 평가하되, 면접 중에는 평가를 드러내지 않습니다.

### 평가 차원

- 문제 해결: 모르는 문제에 접근하는 방식
- 커뮤니케이션: 기술 내용을 명확히 설명하는 능력
- 깊이: 표면적 이해 vs 원리 수준 이해

### 응답 스타일

- 친근하지만 전문적인 톤 ("네, 좋은 답변입니다. 그러면..." / "흥미로운 접근이네요.")
- 후보자가 막힐 때: "이런 관점에서 생각해보시면 어떨까요?"
```

**Q1 (Warm-up):** Spring에서 `@Component`, `@Service`, `@Repository` 어노테이션의 차이가 무엇인가요? 실제 프로젝트에서 이 구분을 어떻게 활용하고 계시는지도 함께 말씀해주세요.

---

### 후보자 답변 1

```
세 개 다 스프링 빈으로 등록해주는 어노테이션입니다. @Component가 가장 일반적이고, @Service는 서비스 레이어, @Repository는 데이터 접근 레이어에 씁니다. @Repository는 DB 예외를 스프링 예외로 변환해주는 기능이 추가로 있습니다.
```

**Claude (면접관):**

네, 정확합니다. `@Repository`의 예외 변환 기능까지 언급해주신 점이 좋습니다.

조금 더 들어가보겠습니다.

**Q2 (Core):** 방금 언급하신 예외 변환에 대해 좀 더 구체적으로 질문드리겠습니다. `@Repository`가 없이 순수 `@Component`로 DAO 클래스를 만들면 실제로 어떤 차이가 발생하나요? 그리고 이 예외 변환이 내부적으로 어떤 메커니즘으로 동작하는지 설명해주실 수 있나요?

---

### 후보자 답변 2

```
@Repository가 없으면 JPA에서 발생하는 예외가 그대로 전파됩니다. 예를 들어 Hibernate의 예외가 서비스 레이어까지 올라오게 되죠. @Repository가 있으면 DataAccessException 계열로 변환되어서 특정 DB 기술에 종속되지 않게 됩니다. 내부 메커니즘은... PersistenceExceptionTranslationPostProcessor가 관여하는 것으로 알고 있는데, 정확한 동작 방식은 잘 기억이 나지 않습니다.
```

**Claude (면접관):**

`PersistenceExceptionTranslationPostProcessor`까지 언급해주셨네요. 맞습니다. 이 BeanPostProcessor가 `@Repository` 빈에 AOP 프록시를 적용하여 persistence 예외를 Spring의 `DataAccessException` 계층으로 변환합니다. 정확한 클래스명까지 기억하고 계신 것은 좋은 신호입니다.

방향을 바꿔서 API 설계 쪽으로 가보겠습니다.

**Q3 (Core):** REST API를 설계할 때 HTTP 상태 코드를 어떻게 사용하시나요? 특히 에러 상황에서의 전략을 설명해주세요. 예를 들어, 사용자가 존재하지 않는 리소스를 요청할 때와 잘못된 데이터를 보낼 때 각각 어떻게 처리하시나요?

---

### 후보자 답변 3

```
200은 성공, 201은 생성 성공, 204는 삭제 성공할 때 씁니다.
에러는 400이 잘못된 요청, 404가 리소스 없음, 500이 서버 에러입니다.
존재하지 않는 리소스 요청은 404, 잘못된 데이터는 400을 반환합니다.
실무에서는 에러 응답 바디에 에러 코드와 메시지를 넣어서 클라이언트가 어떤 에러인지 구분할 수 있게 합니다.
```

**Claude (면접관):**

기본적인 상태 코드 사용은 잘 정리되어 있습니다. 에러 응답 바디에 구조화된 에러 정보를 포함한다는 점도 실무적인 접근이네요.

여기서 한 단계 더 깊이 들어가보겠습니다.

**Q4 (Deep-dive):** 다음 시나리오를 생각해보세요. `POST /orders`로 주문을 생성하는 API에서, 요청은 유효하지만 재고가 부족하여 주문을 처리할 수 없는 경우가 있습니다. 이 경우 어떤 HTTP 상태 코드를 사용하시겠어요? 400인가요, 409인가요, 아니면 다른 코드인가요? 근거와 함께 설명해주세요.

---

### 후보자 답변 4

```
음... 400은 요청 자체가 잘못된 거니까 아닌 것 같고요. 비즈니스 로직상 처리가 안 되는 거니까... 409 Conflict를 쓸 것 같습니다. 현재 리소스 상태(재고)와 요청이 충돌하는 상황이니까요. 아니면 422 Unprocessable Entity도 고려해볼 수 있을 것 같은데, 409가 더 의미가 명확한 것 같습니다.
```

**Claude (면접관):**

좋은 사고 과정입니다. 400을 제외한 논리가 정확하고, 409와 422를 모두 고려한 점이 인상적입니다.

실무에서는 팀마다 다를 수 있는데, `409 Conflict`는 "현재 리소스 상태와의 충돌"을 의미하므로 재고 부족 시나리오에 적합합니다. `422 Unprocessable Entity`는 "요청은 이해했지만 처리할 수 없음"으로 의미론적으로도 맞지만, 409가 좀 더 구체적인 의미를 전달합니다. 정답이 하나만 있는 문제는 아니고, 중요한 것은 팀 내에서 일관된 기준을 갖는 것입니다.

마지막 영역인 데이터베이스로 넘어가겠습니다.

**Q5 (Core):** N+1 쿼리 문제가 무엇이고, Spring Data JPA에서 이를 어떻게 해결하시나요?

---

### 후보자 답변 5

```
N+1은 연관 엔티티를 조회할 때 1번의 메인 쿼리 후 N번의 추가 쿼리가 발생하는 문제입니다.
예를 들어 주문 10건을 조회하면서 각 주문의 회원 정보를 가져올 때 총 11번 쿼리가 실행됩니다.

해결 방법은 세 가지 정도 알고 있습니다.
1. @EntityGraph나 Fetch Join으로 한 번에 가져오기
2. @BatchSize로 IN 절로 묶어서 가져오기
3. DTO 프로젝션으로 필요한 데이터만 직접 쿼리하기

보통은 Fetch Join을 가장 많이 사용하고, 컬렉션이 여러 개면 @BatchSize를 씁니다.
```

**Claude (면접관):**

매우 깔끔한 답변입니다. 세 가지 해결 방법을 비교하고, 상황별 선택 기준까지 제시해주셨네요.

**Q6 (Deep-dive):** Fetch Join으로 `Order`와 `OrderItem`(OneToMany)을 함께 조회할 때 페이징(`Pageable`)을 적용하면 어떤 문제가 발생하나요? 그리고 어떻게 해결하시겠어요?

---

### 후보자 답변 6

```
Fetch Join과 페이징을 함께 쓰면... 메모리에서 페이징을 하는 문제가 있습니다. Hibernate가 경고 로그를 남기는데, 전체 데이터를 메모리로 가져온 다음 애플리케이션에서 잘라내기 때문에 데이터가 많으면 OOM이 발생할 수 있습니다.

해결 방법은 @BatchSize를 사용하거나, 쿼리를 분리해서 먼저 Order ID 목록을 페이징으로 가져온 후 IN 절로 OrderItem을 별도 조회하는 방식이 있습니다.
```

**Claude (면접관):**

정확합니다. `HHH000104: firstResult/maxResults specified with collection fetch; applying in memory` 경고가 바로 그 상황인데요. 쿼리 분리 접근까지 설명해주신 점이 좋습니다.

이것으로 준비한 질문은 마무리되었습니다. 혹시 추가로 하고 싶은 말씀이나 질문이 있으신가요?

### 강점

1. **실무 경험에 기반한 답변**: 단순 이론이 아닌 "보통은 Fetch Join을 쓰고, 컬렉션이 여러 개면 BatchSize" 같은 상황별 판단력이 돋보임
2. **논리적 추론 과정**: HTTP 상태 코드 선택에서 소거법을 사용하며 근거를 제시하는 사고 방식이 우수
3. **핵심 개념의 정확한 이해**: N+1 문제, Fetch Join 페이징 이슈 등 실무에서 자주 만나는 문제에 대한 정확한 이해

### 개선 영역

1. **프레임워크 내부 동작 이해**: `PersistenceExceptionTranslationPostProcessor`를 알고는 있으나 AOP 프록시 적용 과정의 상세 설명이 부족. Spring의 내부 동작 원리를 더 깊이 학습하면 좋겠음
2. **설계 철학 표현**: API 상태 코드 선택에서 좋은 판단을 보였으나, "팀 내 일관성"이나 "API 문서화와의 연계" 같은 거시적 설계 관점을 더 표현하면 시니어로의 성장이 보일 것
