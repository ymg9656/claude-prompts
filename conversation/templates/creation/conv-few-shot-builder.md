# conv-few-shot-builder -- Few-shot 예시 구성

## 사용 시기

- Claude의 응답 형식이나 스타일을 특정 패턴으로 고정하고 싶을 때
- 복잡한 출력 구조를 말로 설명하기보다 예시로 보여주는 것이 효과적일 때
- 도메인 특화된 응답 패턴을 학습시키고 싶을 때
- 기존 프롬프트에 예시를 추가하여 응답 일관성을 높이려 할 때

## 프롬프트

```markdown
당신은 Few-shot 프롬프트 설계 전문가입니다.
다음 요구사항에 맞는 Few-shot 예시 세트를 구성하세요.

### 목표 작업
- 작업 설명: {{TASK_DESCRIPTION}}
<!-- 예: "코드 리뷰 코멘트 생성", "에러 메시지를 사용자 친화적으로 변환" -->
- 입력 형태: {{INPUT_FORMAT}}
<!-- 예: "Python 코드 스니펫", "에러 로그 텍스트", "사용자 질문" -->
- 출력 형태: {{OUTPUT_FORMAT}}
<!-- 예: "마크다운 리뷰 코멘트", "JSON 형식 분석 결과", "단계별 설명" -->

### 출력 패턴 정의
<!-- 예시가 공통으로 따라야 하는 구조를 기술합니다 -->

#### 필수 요소
1. {{OUTPUT_ELEMENT_1}}
<!-- 예: "심각도 표시 (Critical/Warning/Info)" -->
2. {{OUTPUT_ELEMENT_2}}
<!-- 예: "문제 설명 (1-2문장)" -->
3. {{OUTPUT_ELEMENT_3}}
<!-- 예: "수정 제안 코드" -->
{{?ADDITIONAL_ELEMENTS}}

#### 출력 템플릿
```
{{OUTPUT_TEMPLATE}}
```
<!-- 예시들이 공통으로 따르는 출력 구조의 스켈레톤 -->

### 예시 구성 가이드

#### 난이도 분포
- 쉬운 예시 수: {{EASY_COUNT}}
<!-- 권장: 1-2개. 기본 패턴을 보여주는 단순한 사례 -->
- 중간 예시 수: {{MEDIUM_COUNT}}
<!-- 권장: 1-2개. 일반적인 업무 상황 -->
- 어려운 예시 수: {{HARD_COUNT}}
<!-- 권장: 1개. 엣지 케이스나 복잡한 상황 -->

#### 다양성 요구사항
- 포함해야 할 변형: {{VARIATION_REQUIREMENTS}}
<!-- 예: "다른 프로그래밍 언어", "다른 에러 유형", "긍정/부정 사례 모두" -->
- 반드시 포함할 엣지 케이스: {{?EDGE_CASES}}
<!-- 예: "입력이 비어있는 경우", "여러 문제가 동시에 있는 경우" -->

### 기존 예시 (선택)
{{?EXISTING_EXAMPLES}}
<!-- 이미 만들어진 예시가 있다면 여기에 포함하여 스타일 참고 가능 -->

### 컨텍스트 제약
- 최대 토큰 예산: {{?TOKEN_BUDGET}}
<!-- 예: "전체 예시 세트가 2000 토큰 이내" -->
- 사용 위치: {{USAGE_CONTEXT}}
<!-- 예: "시스템 프롬프트 내 포함", "사용자 메시지 앞에 첨부", "별도 컨텍스트로 전달" -->

---

위 요구사항을 바탕으로 다음을 생성하세요:

1. **예시 세트**: 요청된 수만큼의 입력-출력 쌍
   - 각 예시는 `### 예시 N` 형태로 구분
   - 입력과 출력을 명확히 분리 (`**입력:**` / `**출력:**`)

2. **사용 가이드**: 예시 세트를 프롬프트에 삽입하는 방법
   - 전체 삽입용 텍스트 블록 제공
   - "다음 예시들을 참고하여 동일한 형식으로 응답하세요:" 형태의 지시문 포함

3. **검증 지침**: 예시 세트의 품질을 검증하는 방법
   - 새로운 입력으로 테스트할 케이스 2-3개 제안
```

## 품질 기준

- [ ] 예시들이 출력 패턴을 일관되게 따르는가? (형식 통일성)
- [ ] 쉬운 예시부터 어려운 예시까지 난이도가 점진적으로 증가하는가?
- [ ] 각 예시가 서로 다른 유형의 입력을 다루어 다양성이 확보되는가?
- [ ] 엣지 케이스가 최소 1개 포함되어 예외 상황 처리를 보여주는가?
- [ ] 예시의 총 토큰 수가 컨텍스트 윈도우 대비 효율적인가? (전체의 20% 이하 권장)
- [ ] 예시 없이도 지시문만으로 이해 가능하되, 예시가 명확성을 더하는가?
- [ ] 생성된 예시가 실제 업무 데이터와 유사한 현실적인 내용인가?
