# conv-iterative-refinement -- 반복 개선 대화

## 사용 시기

- 한 번의 응답으로 원하는 결과를 얻기 어려운 복잡한 작업일 때
- 초안을 기반으로 점진적으로 품질을 높여가야 할 때
- 요구사항이 대화 중에 구체화되는 탐색적 작업일 때
- 글쓰기, 코드 작성, 설계 등 반복 개선이 자연스러운 작업일 때

## 프롬프트

```markdown
당신은 {{ROLE}}입니다.
다음 작업을 반복 개선(iterative refinement) 방식으로 수행합니다.

### 최종 목표
{{FINAL_GOAL}}
<!-- 예: "프로덕션 수준의 사용자 인증 API 설계", "기술 블로그 포스트 완성" -->

### 현재 상태
{{CURRENT_STATE}}
<!-- 예: "아직 시작 전", "초안이 있음 (아래 첨부)", "2차 리뷰까지 완료" -->

### 현재 산출물 (있는 경우)
{{?CURRENT_ARTIFACT}}
<!-- 현재까지 작성된 코드, 문서, 설계안 등 -->

### 개선 방향
{{IMPROVEMENT_FOCUS}}
<!-- 이번 반복에서 집중할 측면 -->
<!-- 예: "코드 가독성 개선", "엣지 케이스 처리 추가", "문장 간결화" -->

### 반복 개선 프로세스

다음 구조로 응답하세요:

**라운드 {{CURRENT_ROUND}} / 최대 {{MAX_ROUNDS}}**

1. **현재 상태 평가**
   - 현재 산출물의 강점 (유지할 부분)
   - 현재 산출물의 약점 (개선할 부분)
   - 이번 라운드의 우선순위

2. **변경 사항**
   - 변경 내용을 구체적으로 나열
   - 각 변경의 이유 설명
   - 변경 전/후 비교 (해당하는 경우)

3. **개선된 산출물**
   - 전체 산출물을 개선된 버전으로 제시
   <!-- 또는 diff 형태로 변경 부분만 표시 -->

4. **다음 라운드 제안**
   - 추가로 개선 가능한 영역
   - 다음에 집중하면 좋을 측면
   - 현재 완성도 ({{COMPLETION_METRIC}})
   <!-- COMPLETION_METRIC 예: "퍼센트", "5점 척도", "MVP/Beta/Production 등급" -->

### 개선 기준
<!-- 각 라운드에서 참고할 품질 기준 -->
1. {{QUALITY_CRITERION_1}}
<!-- 예: "코드가 SOLID 원칙을 따르는가?" -->
2. {{QUALITY_CRITERION_2}}
<!-- 예: "모든 에러 시나리오가 처리되는가?" -->
3. {{QUALITY_CRITERION_3}}
<!-- 예: "테스트 커버리지가 80% 이상인가?" -->
{{?ADDITIONAL_CRITERIA}}

### 중단 조건
<!-- 반복을 멈추는 기준 -->
- {{STOP_CONDITION_1}}
<!-- 예: "모든 품질 기준을 충족하면 완료" -->
- {{STOP_CONDITION_2}}
<!-- 예: "3회 연속 개선 사항이 미미하면 완료" -->
{{?ADDITIONAL_STOP_CONDITIONS}}
```

## 품질 기준

- [ ] 각 라운드에서 구체적이고 측정 가능한 개선이 이루어졌는가?
- [ ] 이전 라운드의 강점이 유지되고 있는가? (퇴행 없음)
- [ ] 변경 이유가 명확히 설명되었는가?
- [ ] 완성도 평가가 객관적 기준에 근거하는가?
- [ ] 중단 조건이 명확하여 불필요한 반복이 방지되는가?
- [ ] 다음 라운드 제안이 실행 가능하고 구체적인가?
- [ ] 전체 반복 과정에서 일관된 방향성이 유지되는가?
